# M5 — Whole-Campaign World Preload

> 2026-06-11. Engine commit 1cf0420b on `streaming`. The M3 residency cache,
> warmed for the entire campaign at startup: no transition in the game ever
> loads a world from disk again.

## Design

- **`world_preload <mapname>`** (engine/common/model.c) — loads a map through
  the normal `Mod_LoadWorld` path (file, BSP parse, PVS/PHS, texture upload)
  and lets the next call's `Mod_PurgeStudioCache` park it in the residency
  cache. No-op if already resident; refused while a server is active (a
  preload would displace the live world in `mod_known[0]`) or with
  `mod_world_residency 0`.
- **`streampreload.cfg`** — generated by `tools/hlstream_preprocess.py
  --preload-cfg <gamedir>/streampreload.cfg`: one `world_preload <map>` +
  `wait` per connected campaign map (BFS order from the campaign start, so
  the nearest maps warm first if interrupted). One map per frame keeps the
  menu interactive. Derived locally from the user's own data — never
  distributed (constraint: docs/research/00).
- Hooked into `valve/autoexec.cfg` (`exec streampreload.cfg`) so a plain
  launch warms everything behind the menu.

## Measurements (retail HL1, 96 connected campaign maps, RTX 3060)

| Metric | Value |
|---|---|
| Total preload work | **1.78 s** (spread over 96 frames; slowest map c1a3 = 52.7 ms) |
| Resident memory after preload | **~492 MB** total engine pools (~4.8 MB/map; "RAM is free") |
| Map start / changelevel world setup | **~0.5 ms** (`Mod_RestoreCachedWorld`), was ~33 ms |
| First-visit transition (c0a0 chain) | server 11–30 ms, blackout 37–59 ms |
| Revisit transition | server ~11 ms, blackout ~21 ms (unchanged from M4b) |

Every map in the test chain — including the initial `map c0a0` — restored
from the cache. Boot-from-autoexec verified (96/96 preloaded before menu
interaction).

## What dominates transitions now

World load is gone everywhere. First visits still pay per-map entity spawn +
studio model/sound precache (server side) and client resource processing —
that's the 37–59 ms band above, and the next optimization target if we want
first visits at the ~20 ms revisit level:

- studio/sound precache warming (models persist in `mod_known` across maps,
  so each model loads once per session — the cost is concentrated in the
  first transition that uses it);
- svc_resourcelist processing (~6 ms, known from M4a);
- the remaining ~9 ms server-side save/restore loop.

## Caveats

- ~~`vid_restart` drops the residency cache~~ — **corrected after
  verification**: FWGS has no `vid_restart`; resolution/fullscreen changes go
  through `VID_SetMode` with the GL context preserved, and the cache survives
  them (tested: mode change before and during a game, both followed by
  cache-hit restores at the next map load/transition). The cache only drops
  on engine shutdown and on game switches (`CL_UnloadProgs` →
  `Mod_ClearUserData`), where the new game's own autoexec re-warms it — each
  game dir needs its generated cfg + autoexec hook (expansions pass, later).
- Preload is refused during an active server by design; dynamic neighbor
  warming mid-game would need a cache-fill path that doesn't touch slot #0
  (Phase 2 territory, together with stitching).
